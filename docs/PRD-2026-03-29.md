## 1. Project Overview

### 1.1 Background

ComicManager v0.2.0 is a Python desktop application for merging CBZ (Comic Book ZIP) and ZIP files containing comic images into a single CBZ file. The current implementation uses Tkinter for the GUI, which has limitations in visual appeal, responsiveness, and maintainability.

ComicManager Neo is a complete rewrite using modern web technologies for the frontend while preserving all existing core functionality and expanding on it.

### 1.2 Objectives

- Replace Tkinter GUI with a modern, responsive web-based UI (Vue 3 + TailwindCSS + DaisyUI) rendered inside PyWebView
- Replace CLI-only backend with a FastAPI-based local HTTP server
- Maintain all existing features from v0.2.0
- Improve user experience with better visual design and interactions
- Establish a clean, maintainable architecture for future feature expansion

### 1.3 Scope

**In Scope:**
- Full migration of v0.2.0 features to the new tech stack
- CBZ and ZIP file merging
- ZIP image extraction with format filtering
- Smart chapter-based renaming
- Drag-and-drop file management
- Progress tracking with real-time updates
- Settings and configuration management (YAML format)
- Packaging as standalone executable via PyInstaller

**Out of Scope (v1.0):**
- Online/cloud storage integration
- Mobile support
- Plugin system
- Image format conversion
- ComicInfo.xml visual editor
- Batch directory processing

## 4. API Design

### 4.1 Endpoints

```
BASE: http://localhost:{dynamic_port}

# File Queue Management
POST   /api/files              Add files to merge queue (multipart form)
GET    /api/files              Get all queued files with metadata
DELETE /api/files              Remove files by indices (JSON body: {indices: [0,1,2]})
PUT    /api/files/reorder      Reorder files (JSON body: {from: 0, to: 3})
POST   /api/files/clear        Clear all files from queue
POST   /api/files/validate     Validate all queued files

# Merge Operations
POST   /api/merge              Start merge operation
GET    /api/merge/{task_id}/progress  SSE stream for merge progress

# Browse (OS-level file dialogs)
POST   /api/browse/files       Open OS file picker dialog
POST   /api/browse/directory   Open OS directory picker dialog

# Settings
GET    /api/settings           Get current settings
PUT    /api/settings           Update settings (JSON body)

# Formats
GET    /api/formats            Get supported image formats
```

### 4.2 Data Models

```python
# Queued file representation
class QueuedFile(BaseModel):
    path: str              # Absolute file path
    name: str              # Filename
    type: Literal["CBZ", "ZIP"]
    size: int              # File size in bytes
    page_count: int        # Number of images
    valid: bool            # Validation status
    error: str | None      # Validation error if any

# Merge request
class MergeRequest(BaseModel):
    output_filename: str
    output_dir: str
    preserve_metadata: bool = True
    zip_formats: list[str] = ["jpg"]

# Merge progress (SSE event)
class MergeProgressEvent(BaseModel):
    task_id: str
    stage: Literal["validating", "extracting", "merging", "writing", "done", "error"]
    current_file: str | None = None
    current_index: int = 0
    total_files: int = 0
    current_page: int = 0
    total_pages: int = 0
    message: str

# Merge result
class MergeResult(BaseModel):
    success: bool
    output_path: str | None = None
    total_pages: int = 0
    merged_files: list[MergedFileInfo] = []
    errors: list[str] = []

# App settings
class AppSettings(BaseModel):
    last_output_dir: str
    last_input_dir: str = ""
    zip_image_formats: list[str] = ["jpg"]
    theme: str = "light"
    preserve_metadata: bool = True
    auto_increment: bool = True
```

## 8. Non-Functional Requirements

### 8.1 Performance

- File validation should complete within 2 seconds per file (for typical CBZ/ZIP up to 500MB)
- UI must remain responsive during merge operations (background task + SSE)
- Startup time should not exceed 3 seconds

### 8.2 Compatibility

- Windows 10/11 (primary target)
- Python 3.10.8+
- No macOS/Linux support in v1.0 (PyWebView behavior differs significantly)

### 8.3 Security

- Server only binds to 127.0.0.1 (localhost only)
- Path traversal validation on all file inputs
- No credential storage required (no user accounts)
- No network access to external services

### 8.4 Reliability

- Temporary files must be cleaned up on merge completion or failure
- Graceful handling of corrupt ZIP/CBZ files
- No data loss: original files are never modified

## 10. Open Questions

1. **PyWebView DnD**: PyWebView's file drop support varies by platform. Need to verify HTML5 drag-and-drop works correctly on Windows with PyWebView 6.x. Fallback: file picker button only.

2. **SSE vs WebSocket**: SSE is simpler for progress updates. WebSocket adds complexity. Decision: start with SSE, switch to WebSocket only if bidirectional communication is needed later.

3. **PyInstaller single-file vs single-directory**: Single-file is more convenient for distribution but has slower startup. Decision: provide both options via spec file.

4. **Bundled frontend assets**: For PyInstaller builds, the built Vue frontend (dist/) should be served directly by FastAPI as static files, eliminating the need for a separate dev server in production.
