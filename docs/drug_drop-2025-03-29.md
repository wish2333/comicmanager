# pywebview 拖拽文件读取 - 完整实现参考

基于 ComicManager Neo 项目中实际调试成功的实现，记录 pywebview (Windows/EdgeChromium) 下拖拽文件读取的完整流程和踩坑经验。

## 核心结论

浏览器标准 HTML5 DnD API 中的 `File` 对象**无法获取完整文件路径**（`file.path` 是 Electron 专有 API）。pywebview 基于 EdgeChromium (WebView2)，提供了内部机制绕过此限制，但必须通过其 DOM API 而非标准 `window.addEventListener` 来使用。

## 完整流程

### 第 1 层：前端 - 视觉反馈（window 级事件监听）

```typescript
// composables/useFileDrop.ts
const isDragging = ref(false)
let dragCounter = 0

// 在 window 上注册 dragenter/dragover/dragleave 用于显示拖拽覆盖层
window.addEventListener('dragenter', (e) => {
  e.preventDefault()
  dragCounter++
  isDragging.value = true
})
window.addEventListener('dragover', (e) => e.preventDefault())
window.addEventListener('dragleave', () => {
  dragCounter--
  if (dragCounter <= 0) { dragCounter = 0; isDragging.value = false }
})

// window 上的 drop 事件用于触发文件路径获取
window.addEventListener('drop', async (e) => {
  e.preventDefault()
  isDragging.value = false
  // 轮询 Python 端获取文件路径（见第 3 层）
  const result = await window.pywebview.api.get_dropped_files()
  if (result.success && result.data.length > 0) {
    addFiles(result.data)
  }
})
```

**关键点**：
- 必须在 `window` 上监听，不能只在 `document` 上
- `e.preventDefault()` 是必须的，否则浏览器会尝试导航到拖入的文件
- 使用 `dragCounter` 而非简单的布尔值来处理子元素触发的 dragleave 事件

### 第 2 层：Python - 通过 pywebview DOM API 注册 drop 处理器

```python
# src/js_api.py
def setup_drag_drop(api: JsApi) -> None:
    import webview
    from webview.dom import DOMEventHandler

    def on_loaded():
        window = webview.windows[0]

        # 关键：通过 pywebview 的 DOM API 注册 drop 事件处理器
        # 这会自动完成三件事：
        # 1. 在 _dnd_state['num_listeners'] +1（启用 CoreWebView2File 路径提取）
        # 2. 在 document 上注入 JS 事件处理器，调用 pywebview._jsApiCallback
        # 3. 将 'document' 注册到 window.dom._elements 中
        doc = window.dom.document
        handler = DOMEventHandler(api._on_drop, prevent_default=True)
        doc.on("drop", handler)

    # 必须在 window loaded 事件后注册
    window = webview.windows[0]
    window.events.loaded += on_loaded
```

**关键点**：
- `doc.on("drop", handler)` 内部会自动调用 `_dnd_state['num_listeners'] += 1`
  - **不要手动再 `_dnd_state['num_listeners'] += 1`，否则会重复计数**
- 必须用 `DOMEventHandler(callback, prevent_default=True)` 包装回调
  - `Element.on()` 不接受 `prevent_default=True` 关键字参数（TypeError）
  - 正确用法是传入 `DOMEventHandler` 对象
- 必须在 `window.events.loaded` 回调中执行，否则 `window.dom.document` 不可用
- `doc.on("drop", ...)` 内部已自动注入调用 `pywebview._jsApiCallback` 的 JS 代码
  - **不要再手动注入额外的 `document.addEventListener('drop', ...)` 调用 `_jsApiCallback`**
  - 否则 pywebview bridge 被调用两次，文件会被导入两次

### 第 3 层：Python - 回调中读取 pywebviewFullPath

```python
def _on_drop(self, event: dict) -> None:
    """pywebview DOM 事件系统调用的回调"""
    files = event.get("dataTransfer", {}).get("files", [])
    paths: list[str] = []
    for f in files:
        # pywebview 在 util.py 中为每个文件注入了 pywebviewFullPath
        full_path = f.get("pywebviewFullPath")
        if full_path:
            paths.append(full_path)

    if paths:
        # 存入缓冲区，等待前端通过 API 获取
        with self._drop_lock:
            self._dropped_paths.extend(paths)
```

**关键点**：
- `event` 是一个 `dict`，不是 JS 的 `Event` 对象
- `event["dataTransfer"]["files"]` 中的每个文件对象会有 `pywebviewFullPath` 字段
- `pywebviewFullPath` 由 pywebview 的 `util.py` 在 `pywebviewEventHandler` 处理中自动注入
- 注入逻辑：匹配 `_dnd_state['paths']` 中的 `(basename, full_path)` 对，按文件名配对

### 第 4 层：前端 - 轮询获取文件路径

```typescript
// 前端 window drop 事件触发后，轮询 Python 端
for (let attempt = 0; attempt < 30; attempt++) {
  const result = await window.pywebview.api.get_dropped_files()
  if (result.success && result.data && result.data.length > 0) {
    dropCallback(result.data)
    return
  }
  await new Promise((r) => setTimeout(r, 100))  // 100ms 间隔
}
```

**关键点**：
- 必须轮询，不能直接在 drop 事件中同步获取
- pywebview 的 FilesDropped 消息处理是异步的：
  1. JS `drop` 事件触发 -> `pywebview._jsApiCallback` 被调用
  2. WebView2 桥接发送 `FilesDropped` 消息 + `dataTransfer.files`
  3. Python `on_script_notify` 收到消息，读取 `CoreWebView2File.Path`
  4. 路径存入 `_dnd_state['paths']`
  5. JS API 回调到达 Python `util.py`，匹配文件名，注入 `pywebviewFullPath`
  6. `_on_drop` 回调被调用
- 步骤 2-5 是异步的，JS 端 drop 事件处理函数返回后才完成
- 因此前端需要轮询 `get_dropped_files()` 等待 Python 端处理完成

## pywebview 内部机制详解

### 事件流时序

```
用户拖入文件
  |
  v
浏览器原生 dragenter/dragover/dragleave (window 级)
  -> 前端显示拖拽覆盖层
  |
  v
用户释放文件
  |
  v
pywebview 注入的 document drop handler 触发
  -> pywebview._jsApiCallback('pywebviewEventHandler', {event, nodeId:'document'})
  |
  v
[WebView2 桥接层] api.js 检测到 drop event 参数
  -> chrome.webview.postMessageWithAdditionalObjects('FilesDropped', dataTransfer.files)
  |
  v
[Python] edgechromium.py on_script_notify 收到 'FilesDropped'
  -> 读取 CoreWebView2File.Path 获取真实路径
  -> 存入 _dnd_state['paths'] += [(basename, full_path), ...]
  |
  v
[Python] edgechromium.py on_script_notify 收到 js bridge 调用
  -> func_name == 'pywebviewEventHandler'
  -> 检查 _dnd_state['num_listeners'] > 0 (否则直接 return!)
  -> 匹配文件名，注入 file['pywebviewFullPath'] = full_path
  -> 清理 _dnd_state['paths']
  -> 调用 doc.on("drop") 注册的 Python 回调 (_on_drop)
  |
  v
[Python] _on_drop 将路径存入 _dropped_paths 缓冲区
  |
  v
[JS] window 级 drop 事件触发 (事件冒泡到 window)
  -> 前端轮询 window.pywebview.api.get_dropped_files()
  -> 获取到文件路径，调用 addFiles()
```

### 关键内部数据结构

```python
# webview/dom/__init__.py
_dnd_state = {
    'num_listeners': 0,   # drop 处理器数量，>0 时才处理 FilesDropped
    'paths': []           # [(basename, full_path), ...] 临时存储
}
```

### _dnd_state['num_listeners'] 的作用

在 `edgechromium.py on_script_notify` 中：

```python
if return_value == '"FilesDropped"':
    if _dnd_state['num_listeners'] == 0:  # 关键检查！
        return                              # 直接丢弃，不处理文件路径
```

- 如果 `num_listeners == 0`，即使文件被拖入，路径也不会被提取
- `doc.on("drop", callback)` 会自动 +1，`doc.off("drop", callback)` 会自动 -1
- 手动 `_dnd_state['num_listeners'] += 1` 会与 `doc.on()` 的自动递增重复

## 踩坑记录

### 坑 1：Element.on() 不支持 prevent_default 关键字参数

```python
# 错误 - TypeError: Element.on() got an unexpected keyword argument 'prevent_default'
doc.on("drop", api._on_drop, prevent_default=True)

# 正确 - 使用 DOMEventHandler 包装
from webview.dom import DOMEventHandler
handler = DOMEventHandler(api._on_drop, prevent_default=True)
doc.on("drop", handler)
```

### 坑 2：重复注册导致文件导入两次

```python
# 错误 - 文件会被导入两次
_dnd_state['num_listeners'] += 1           # 手动 +1
doc.on("drop", handler)                    # 又自动 +1 -> num_listeners = 2

# 同时手动注入 JS 也会重复调用 pywebview._jsApiCallback
window.evaluate_js("""
    document.addEventListener('drop', function(e) {
        window.pywebview._jsApiCallback(...)  // 第二次调用！
    });
""")

# 正确 - 只用 doc.on()，它内部处理了所有事情
doc.on("drop", handler)  # 自动 +1，自动注入 JS handler
```

### 坑 3：File.path 不可用（Electron 专有）

```javascript
// 错误 - pywebview 中 file.path 始终是 undefined
const filePath = file.path  // undefined

// 正确 - 必须通过 pywebview 的 CoreWebView2File 机制获取路径
// 通过 doc.on("drop") + pywebview._jsApiCallback -> pywebviewFullPath
```

### 坑 4：前端必须在 window 级监听 drop 事件

```javascript
// 不够 - 只在 document 上注册，pywebview 内部的 prevent_default
// 会阻止事件冒泡到 window，前端无法感知 drop 发生
document.addEventListener('drop', handler)

// 正确 - 在 window 上监听（事件冒泡目标）
window.addEventListener('drop', handler)
```

### 坑 5：必须在 loaded 事件后注册

```python
# 错误 - window.dom 可能未初始化
doc = webview.windows[0].dom.document
doc.on("drop", handler)

# 正确 - 等待页面加载完成
def on_loaded():
    doc = webview.windows[0].dom.document
    doc.on("drop", handler)
window.events.loaded += on_loaded
```

### 坑 6：WebDialog.FOLDER / OPEN 替代已废弃常量

```python
# 已废弃 - 产生 deprecation warning
webview.OPEN_DIALOG
webview.FOLDER_DIALOG

# 正确 - 使用 FileDialog 枚举
webview.FileDialog.OPEN
webview.FileDialog.FOLDER
webview.FileDialog.SAVE
```

### 坑 7：create_file_dialog 参数是 directory 而非 folder

```python
# 错误 - TypeError: unexpected keyword argument 'folder'
webview.windows[0].create_file_dialog(dialog_type=webview.FileDialog.FOLDER, folder="/path")

# 正确 - 参数名是 directory
webview.windows[0].create_file_dialog(dialog_type=webview.FileDialog.FOLDER, directory="/path")
```
